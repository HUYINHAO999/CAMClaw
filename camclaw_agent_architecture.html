1. 总体架构

读图方式：上方是普通 UI 到 CAM 命令入口的主链路；下方是 Agent 旁路。轻量 UI 交互不进入 Agent Gateway，Agent 也不绕过 Console 直连 Core。Trace 是横切能力，贯穿从 LLM 到客户端反馈的全过程。

UI / CAM 主链路：轻量交互留在 UI，业务命令进入 Console

Qt UI / Interaction Layer菜单、功能树、参数面板、Viewport；悬停、展开、视角旋转等轻量交互在此结束
CAM Command Console统一承接会改变 CAM 状态的命令：创建工序、改参数、生成刀轨等
Action Dispatcher / Command Bus命令调度、撤销、业务服务、刀轨服务
CAM Core真实 CAM 业务逻辑、工序、刀具、刀轨计算
Canonical Model / Repository稳定对象 ID、真实几何、刀轨、预览网格
↗Adapter 回接 CAM Command Console，不绕过 Console 直连 CAM Core

Agent 旁路扩展：只开放受控能力，经过安全边界后接回 CAM Command Console

Prompt / MCP / Script自然语言、MCP 分组工具、受控 cam.* 脚本
Agent Planner理解意图，结合 Manifest / RAG / Template 选择 action
Agent Action Request标准工单：request_id、trace_id、action_id、args、source
Agent Action GatewayRegistry、Schema、Validator、Human in loop
Action Adapter / Facade把受控 action 映射到 CAM Command Console
Manifest / RAG / Template支撑 Agent Planner，不直接执行业务
Schema / Pydantic约束 Action Request 和参数草案
Logs / Undo / Audit命令日志和撤销体系可与 trace_id 串联
Trace Service 是横切链路，不属于单独某一层 覆盖 Prompt / MCP / Script、LLM plan、Action Request、校验结果、用户确认、Adapter 调用、CAM 执行结果、UI 反馈和错误信息。
已有 UI Agent 规划 Trace / 标准请求 Agent 安全边界与 Adapter 既有 CAM 系统

2. Action 执行链路

这是 Agent 旁路的运行时流程。普通 UI 轻量交互不走这里；Agent、MCP、受控脚本要改变 CAM 状态时，先变成标准 Action Request，再通过 Adapter 接回 CAM Command Console。

1

Trace 开始

从 Prompt / MCP / 脚本入口创建 trace_id。

2

Agent 规划

LLM 根据 Manifest 选择 action，结合 RAG 和模板生成草案。

3

Action Request

形成标准工单：request_id、trace_id、action_id、args、source。

4

Gateway 调度

Agent Action Gateway 查 Registry 和 Schema。

5

Validator

判断当前 CAM 状态：可执行、阻塞、缺参数、需确认。

6

Human in loop

按 low / high 风险使用不同确认体验。

7

Adapter / Facade

把受控 action 映射到 CAM Command Console。

8

CAM 命令执行

由 Console 进入 Command Bus、CAM Core、Undo、日志和 UI 反馈。

5. 参数 Schema 扩展路径

Source of Truth

  • schema/*.json 或 schema/*.yaml 是参数定义权威来源。
  • 新增刀具参数、工序参数、默认值、校验规则时，优先改 schema。
  • 必要时升级 schema_version。

Schema 驱动的下游

  • Python Agent 层：生成 Pydantic 模型。
  • C++/Qt Core：生成或同步 ParameterValidator。
  • UI：参数面板按 schema 渲染。
  • Agent：参数草案和模板实例化按 schema 约束。

6. 架构边界和安全规则

LLM 负责

理解意图、选择 action、生成参数草案、解释推荐原因。

LLM 不负责

不直接调用 CAM Core，不绕过 Gateway / Adapter，不执行任意 Python / 系统命令。

Validator 负责

每个 action 自己判断当前状态能不能执行，例如工序是否存在、刀具是否绑定。

Agent Gateway 负责

只管理 Agent/MCP/脚本自动化入口，不接管鼠标悬停、面板展开、视角旋转等轻量 UI 交互。

Human in loop

V1 区分 low / high，不同风险使用不同确认体验。

稳定引用

Agent 不能用 UI 显示名、树路径、模糊文字直接执行，必须解析到 object_id。

Adapter / Facade

把受控 action 映射到 CAM Command Console，避免 Agent 绕过命令调度直连 Core。

Trace Service

横切记录从 LLM 到客户端反馈的全过程，并可与命令日志通过 trace_id 串联。
