CAMClaw Agent Tool Schema 架构图

已被后续方案替代 这页记录的是中间演进方案。当前采用的最终方向是 SemanticIntentPlan：LLM 只做语义解析，Qt/C++ 直接执行 semantic plan。请优先查看 semantic-intent-refactor-uml.html。

目标不是马上上 MCP，而是先把 Agent 层术语和数据契约理清：LLM 看到的是 skill/tool，Python 用 Pydantic 验证，C++ 执行的是 command/service。

现在的问题 command_id/schema_id/inputs 同时承担 LLM tool、schema、C++ 路由和 UI console 名称。
目标结构 skill -> tool -> Pydantic schema -> validated plan -> C++ command。
迁移策略 Python 先强类型化，Qt/C++ 暂时兼容旧 draft，随后删除兼容层。

三层命名

层 给谁看 例子 职责
Skill LLM / Planner operation.skill.v1, toolpath.skill.v1 能力包，描述有哪些 tools、什么时候用、输入输出规则。
Tool LLM 输出 operation.create, toolpath.set_visibility Agent 可调用动作。短期 1:1 映射 C++ command，长期可 1:N。

HumanInLoop

HumanInLoop 不补脏字段，不替执行层猜意图。它处理不确定性，并把计划状态切到 awaiting_input。

stateDiagram-v2
  [*] --> draft_requested
  draft_requested --> pending_review: valid plan
  draft_requested --> awaiting_input: missing/ambiguous intent
  awaiting_input --> pending_review: user clarified
  pending_review --> confirmed: user confirms
  pending_review --> rejected: user rejects
  confirmed --> executing
  executing --> completed
  executing --> failed
  completed --> [*]
  failed --> [*]
  rejected --> [*]
          
{
  "status": "awaiting_input",
  "clarification": {
    "reason": "missing_operation_type",
    "message": "请选择要隐藏哪类刀轨",

      { "label": "型腔铣", "value": "pocket" },
      { "label": "钻孔", "value": "drilling" },
      { "label": "平面铣", "value": "finishing" }
    ]
  }
}

渐进迁移

阶段 改动 保留兼容 完成标志
Phase 1 Python backend 引入 Pydantic、Skill Registry、ToolCall 格式。 Python 输出仍转成旧 Qt draft：command_id/schema_id/inputs。 LLM raw action 不再绕过 Pydantic。
Phase 2 Qt/C++ 增加 ToolDraft / CommandAdapter。 旧 draft 只作为兼容输入。 C++ 核心不再依赖裸 map<string,string>。
Phase 3 删除旧 command_id 兼容层。 无。 全链路使用 skill/tool/schema 命名。
两阶段 Planner 触发条件 当 skill 数量 > 10，或 system prompt > 4000 tokens 时，从单阶段 prompt 切换为 SkillRouter + ToolPlanner 两阶段。
