CAMClaw Semantic Intent 重构方案

这版不再让 LLM 输出具体执行事件。LLM 只解析用户自然语言，输出强类型语义意图；Qt/C++ 传统 CAM 后端负责目标解析、特征识别、工序选择、参数策略、刀具选择、刀轨计算和 UI 刷新。

LLM 职责 解析自然语言为 SemanticIntentPlan，例如创建工序、修改参数、隐藏刀轨。
Python 职责 用 Pydantic 校验请求、LLM raw JSON、意图 schema、输出 draft。
Qt/C++ 职责 执行传统 CAM 业务：解析目标、特征识别、创建工序、选刀、计算刀轨、刷新 UI。
重构点 当前问题 目标设计
LLM 输出 command_id/schema_id/inputs 混合了 LLM tool 和 C++ command。 输出 SemanticIntentPlan，只表达语义，不指定 C++ 接口。
强类型校验 主要靠手写 parse 和 map<string,string>。 Python 端 Pydantic 校验每种 intent；C++ 端逐步强类型 command。
Target current_selection / operation_type / all / same_as_previous 混杂。 统一为 objects / query / ref 三种 target。
多意图 容易产生组合型大意图，例如 edit_operation_and_recompute。 拆成顺序意图：edit_operation + regenerate_toolpath。
HumanInLoop 主要在执行时解决目标不唯一。 作为通用 interrupt：缺字段、目标不唯一、语义冲突都进入 awaiting_input。

Target 只保留三种互斥类型：明确对象、条件查询、引用前一步结果。当前选择由 Qt 请求上下文提供，不作为 target kind。

kind 运行时落点

kind 不是文档概念，它在 Python 端作为 Pydantic discriminated target 被校验，在 Qt 端被解码成 SemanticTargetKind，最终在 SemanticIntentExecutor::resolveTargetIds 里决定目标解析分支。

kind Python 校验 Qt/C++ 解码 执行含义
objects ObjectsTarget 要求 object_ids 非空。 SemanticTargetKind::Objects 直接把这些对象作为目标，通常来自当前选择或 HumanInLoop 的用户选择。
query QueryTarget 要求 object_type、scope，matching 时必须有 filters。 SemanticTargetKind::Query 由 Qt 在当前 Repository/树模型里查询目标；结果不唯一时进入 HumanInLoop。
ref RefTarget 目前只允许 previous.primary_object_ids。 SemanticTargetKind::Ref 多 intent 顺序执行时，后一 intent 复用前一 intent 的结果，避免重新猜目标。

迁移步骤

阶段 LLM / Python Qt / C++ 验收标准
Phase 1 新增 Pydantic SemanticIntentPlan，LLM 输出 intent，不再输出 command。 Qt 新增 SemanticPlanDraftJsonCodec 和 SemanticIntentExecutor，CAMClaw 弹窗直接执行 semantic plan。 LLM raw JSON 经过 Pydantic，Qt 不再把 Agent 草案转回旧 command_id。
Phase 2 增加 awaiting_input draft 响应，缺字段直接进入 HumanInLoop。 新增 IntentDispatcher、TargetResolver(objects/query/ref)。 复杂任务支持多个 intent 顺序执行。
Phase 3 清理旧 command prompt，保留 semantic contract。 清理历史 AgentPlanDraft agent 路径，C++ 核心转强类型 command/service。 Qt/C++ 不再消费裸 map<string,string> 业务参数。
