总览

当前 Qt 迁移已经具备主窗口、绘图区、树、Agent 审核弹窗、后端草案请求，以及一条从 SkillRuntime 到 ActionGateway 再到 BrowserConsole 的命令执行链路。但普通 CAM 的建模还很薄，且工序/刀轨业务仍被放在了树控制台里。

Qt UI 层

CamMainWindow 聚合窗口、当前这一棵树、绘图区、Agent 弹窗入口，以及当前执行依赖。

执行层

ActionGateway 目前只路由到一个 BrowserConsole，业务服务接口还没有拆出来。

关键偏差

BrowserConsole 现在直接承担了创建工序/生成刀轨业务。 这和“UI 交互入口”定位不一致。

当前问题

核心问题

BrowserConsole 目前承担了不该承担的工序/刀轨业务： createRoughingOperation 和 generateToolpath 都在它内部。 这会让“UI 交互入口”和“CAM 业务服务”的边界混在一起。

  • ActionGateway 只持有一个 BrowserConsole，还没有把工序、刀轨等业务动作委托给独立 Service。
  • CamMainWindow 聚合了 UI、Repository、Gateway、Runtime、Executor、DraftService 等过多职责。
  • Repository 当前对象模型较薄，主要是 object_id、ObjectType、display_name，工序参数、刀具、刀轨状态还没有形成完整领域模型。
  • 普通 CAM UI 还缺少工序选择、几何体/特征差异化 mock、工序编辑、右键菜单、自动计算/删除刀轨等基础工作流。
  • 当前代码里树结构还直接落在 CamMainWindow / QTreeWidget 附近，尚未抽象出 BrowserTree、BrowserTreeItem、FrameItem、OperationItem 等 UI 对象。
  • Agent 能产出结构化草案，但当前体验容易显得像“模板回答”，需要建立在更可信的 CAM 基础能力之上。

Prompt 驱动案例池

这里记录的是“用户用一句话驱动 CAM 完成一串具体动作”的候选案例。 这些案例不是要求马上实现 LLM，而是用来反推普通 CAM、服务接口、树 item 行为和自动化测试。

批量创建工序

Prompt：帮我创建 10 个钻孔工序。 期望：系统批量创建 10 个可见的 Operation 节点，命名可区分，树上能看到每个工序。

打开并修改工序

Prompt：帮我打开这个工序界面，把安全高度改成 5mm，然后重新计算。 期望：定位当前工序，打开编辑界面，修改参数，标记刀轨需重算，再触发计算。

不满意后重算

Prompt：当前刀轨我不满意，改用更小的 stepover 重新生成。 期望：找到当前工序参数，减小横向步距，删除或覆盖旧刀轨，生成新的刀轨预览。

可转成测试的验收点

  • 批量创建：输入数量为 10 时，Repository 中出现 10 个 Operation，BrowserTree 中出现 10 个对应 OperationItem。
  • 批量创建：每个 Operation 都有稳定且不同的 object_id，显示名重复也不能影响后续定位。
  • 工序工厂：创建不同工序时走 OperationFactory / OperationCatalog，而不是在 UI 或 Agent 里硬编码几十个分支。
  • 打开工序：从 prompt 或右键菜单定位到当前 OperationItem 后，能打开同一个工序编辑界面。
  • 修改参数：参数变更通过 OperationService 写入工序参数集，并让既有刀轨变成过期或需重算状态。
  • 重算刀轨：更小 stepover 重新计算后，旧 Toolpath 被替换或标记历史，新 Toolpath 节点和视图区预览同步更新。
  • 推荐工序：帮我选择合适的工序加工这个型腔 不应在 Qt 里按中文关键词硬编码，而应依赖 Feature、OperationCatalog、ToolLibrary 和 Planner/Recommendation 的结构化结果。
  • 信息不足：没有明确目标或当前选择不合法时，系统停止并询问，而不是默认对错误对象创建工序。

下一步目标拆分

目标不是马上把 LLM 做得更聪明，而是先把普通 CAM 的领域入口拆清楚： 人工操作和 Agent 操作都应该进入同一套结构化命令。Console 更接近 UI 对应的交互入口， 工序、刀轨等领域动作应继续下沉到可复用的 Service。当前只有一棵树，未来如果出现多棵树， BrowserConsole 下方需要重构出 BrowserTree 和 BrowserTreeItem 层。

拆分原则

  • BrowserConsole 是和浏览器区域 UI 对应的交互入口，可以协调当前树，也为未来多棵树预留位置。
  • BrowserTree 维护一棵具体的树；当前只有一棵树，未来如果有多棵树，需要把树状态从 CamMainWindow 进一步迁移出来。
  • BrowserTreeItem 维护树节点行为，下面可以有 FrameItem、OperationItem、ToolPathItem 等具体 item。
  • 例如“右键计算刀轨”应从 UI 菜单进入 BrowserConsole，传到当前 BrowserTree，再分发到选中的 OperationItem 内部执行。
  • BrowserConsole 不直接沉淀工序/刀轨业务逻辑，而是调用 OperationService 和 ToolPathService。
  • OperationService 负责创建、编辑、删除工序；创建工序不强制要求先选择几何体。
  • ToolPathService 负责计算、删除、刷新刀轨预览。
  • ActionGateway 继续接收结构化命令，后续可把 UI 事件入口和 Service 调用关系整理得更清楚。
  • 未来 FeatureRecognitionService 根据几何/特征给出推荐工序，但短期不硬限制“每种几何只能创建匹配工序”。
