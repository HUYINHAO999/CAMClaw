CAMClaw 当前代码架构图

这页是按当前代码整理出的架构快照。重点不是画所有依赖，而是把当前可运行链路讲清楚：人工 CAM 和 LLM CAM 最后都会回到 BrowserConsole，再由 CAM services 改 Repository 和刷新 UI。

总览 运行时主链路 Qt / C++ 当前结构 Python / LLM 后端 对象模型 已支持用例 典型案例数据流 边界规则 核心文件

1. 总览

当前是第三版架构。它保留传统 Qt CAM 软件原有的人工操作链路，同时把 LLM 限定在语义解析和 Pydantic 校验边界内；真正 CAM 执行仍在 Qt / C++ 客户端。

第一版：所有入口先过统一适配层 已放弃

设想是无论用户点击、Prompt、脚本，未来都先进入一个 Adapter，把输入转成同一种格式，再进入 Console。

UI 点击用户操作
Prompt / Script自动化入口
Adapter统一格式
  • 问题 1：人工 UI 本来可以直接转发给 BrowserConsole，却被迫多走一层。
  • 问题 2：传统软件的原有结构被打散，普通按钮逻辑也要经过 Agent 化抽象。
  • 问题 3：轻量 UI 事件和自动化任务混在同一入口，边界变虚。
  • 问题 4：后续排查 bug 时，很难判断问题来自 UI、Adapter 还是 Console。
→

第二版：LLM 输出客户端要调用的接口 已放弃

这一版收紧了 LLM 的行为边界，但仍然让模型输出类似 command / tool / schema 的执行指令，客户端再按指令调用接口。

LLM决定接口名
JSON Command带 action 和参数
Client按 command 执行
  • 问题 1：未来客户端接口变多后，LLM 容易选错接口或拼错字段。
  • 问题 2：很多能力只在客户端本地，例如刀具库、特征识别、当前树状态、Repository、HumanInLoop。
  • 问题 3：LLM 感知不到完整 CAM 状态，却被要求决定具体执行接口，职责过重。
  • 问题 4：接口名一旦泄露给 LLM，后续重构 C++ 接口会牵动 prompt 和 schema。
  • 问题 5：容易重新滑向“LLM 直接驱动客户端内部实现”，扩展后维护成本高。
→

第三版：LLM 只输出语义意图，客户端执行 CAM 当前代码

当前版本把 LLM 当作语义解析器，不让它决定 Qt/C++ 具体接口。Python 后端返回 SemanticIntentPlan，Qt 客户端解析后进入现有 BrowserConsole 和 CAM services。

  • Prompt 只新增一条后端规划链路，确认执行后回接 BrowserConsole。
  • 刀具选择、特征识别、目标解析、HumanInLoop 都留在客户端 CAM 执行侧。
  • LLM 输出的是 machine_feature、create_operations、edit_operation 等语义 intent。
  • 未来重构 C++ service 或 UI 结构，不必要求 LLM 学会内部接口名。

2. 运行时主链路

读图方式：上方是人工 CAM 操作；下方是 LLM CAM 操作。两条链路最终都进入 BrowserConsole，后面共用同一套 CAM service 和 Repository。

人工 CAM 操作

CamMainWindow接收顶部工序按钮、树双击、右键菜单
BrowserConsole统一调度 create / calculate / delete / visibility
CAM ServicesOperationService、ToolPathService 执行业务
Repository保存 Feature / Operation / Toolpath 对象
BrowserTree / CamViewport刷新树、显示或隐藏刀轨预览
Agent 确认执行后回接 BrowserConsole，不绕过 Console 直连 Service

LLM CAM 操作

AgentReviewDialog输入 prompt，生成草案，用户确认执行
HttpAgentDraftService客户端发起 POST /v1/agent/plan，并接收响应 body
Python Backend后端调用 LLM + Pydantic 校验，返回 JSON 草案
SemanticPlanDraftJsonCodec客户端解析后端 JSON，形成 SemanticPlanDraft
SemanticIntentExecutor顺序执行 intents，必要时触发 HumanInLoop
BrowserConsole进入人工 CAM 相同执行入口
Repository / UI更新 CamObject，并刷新 BrowserTree / CamViewport
HumanInLoopService目标不唯一时让用户选择，不绑定具体业务。
FeatureRecognitionService当前用字符串 mock 识别：孔 / 型腔 / 平面。
ToolSelectionPolicy解析 larger_available_tool / smaller_available_tool。
trace_id贯穿草案生成、执行结果和 trace_events。

3. Qt / C++ 当前结构

这里画当前代码的组合关系。Console 只保留 BrowserConsole；Operation / ToolPath / Feature 都是 service，不叫 console。

CamMainWindow

  • Qt 侧组合根
  • 创建 Repository
  • 创建 BrowserConsole
  • 创建 SemanticIntentExecutor
  • 持有 HttpAgentDraftService
BrowserTree当前是 QTreeWidget 包装层；未来可拆 BrowserTreeItem / OperationItem / FrameItem。
CamViewport显示几何 mock、工序预览、刀轨显隐状态。
AgentReviewDialog草案审核弹窗，展示 prompt、草案、参数、结果、trace。
BrowserConsole当前唯一 Console。负责把 UI 事件和 Agent intent 路由到 CAM services。
SemanticIntentExecutor执行 machine_feature、create_operations、edit_operation、regenerate_toolpath、set_toolpath_visibility。
HumanInLoopService当目标 query 命中多个对象时，弹窗让用户澄清。
OperationService通过 OperationFactory / OperationCatalog 创建工序。
ToolPathService按 operation_id 生成或更新对应 toolpath。
FeatureRecognitionService根据所选 feature mock 出匹配 operation_type。
ToolSelectionPolicy在 C++ 侧选择可用刀具，不让 LLM 直接决定最终刀号。

4. Python / LLM 后端

Python 后端是语义解析服务。它不创建工序、不生成刀轨、不访问 Qt Repository。

POST /v1/agent/plan

server.py接收 JSON 请求，调用 AgentPlanRequestModel 校验
planner.py组装 response_contract，调用 LLM
llm_client.pyOpenAI-compatible chat completions
semantic_schemas.pyPydantic 校验 SemanticIntentPlan
draft JSON返回 schema_version、intents、trace_id、status

AgentPlanRequest

  • trace_id：一次 Agent 请求的追踪 ID
  • user_request：用户 prompt
  • target_object_id：Qt 当前选择对象 ID
  • target_display_name：当前选择对象显示名
  • rejection_reason：拒绝草案后的再生成反馈

SemanticIntentPlan

  • schema_version = "1"
  • intents[]：有序动作序列
  • 支持多意图：先 edit_operation，再 regenerate_toolpath
  • LLM 输出必须通过 Pydantic 校验

5. 当前对象模型

当前 Repository 保存的是 CamObject。真实几何和刀轨仍是 mock，但粒度已经按 object_id / operation_id / toolpath_id 走，而不是按 operation_type 共用一套刀轨。

Repository

  • save
  • update
  • remove
  • objectsByType

Feature

  • 孔组：匹配 drilling
  • 型腔：匹配 pocket
  • 平面：匹配 finishing

Operation

  • 拥有自己的 operation_id
  • 记录 operation_type
  • 记录 tool_id、stepover 等参数
  • toolpath_status 表示是否已计算

Toolpath

  • 拥有自己的 toolpath_id
  • parent_object_id = operation_id
  • 按 operation_id 生成，不按 operation_type 共享
  • visible 控制显隐

7. 典型案例数据流

这一节用三个具体问题说明任务拆解边界：LLM 到 SemanticIntentPlan 为止；Qt 解码、解析目标、调用 BrowserConsole；BrowserConsole 再进入 CAM services。

案例 1：选择孔组后输入“帮我加工这个工件” 目标：根据当前选择的几何特征，自动创建钻孔工序，并生成钻孔刀轨。
1. Qt 收集上下文

用户在 BrowserTree 选中孔组，打开 AgentReviewDialog。

2. 请求 Python

HttpAgentDraftService 发送 prompt、target_object_id、target_display_name。

3. LLM 语义解析

LLM 不判断最终 CAM 细节，只输出 machine_feature intent。

4. Qt 解码草案

SemanticPlanDraftJsonCodec 解成 SemanticPlanDraft，用户确认。

5. 执行 intent

SemanticIntentExecutor 把 objects target 交给 BrowserConsole.machineFeature。

6. CAM 执行

FeatureRecognitionService 识别 drilling，OperationService 创建工序，ToolPathService 生成刀轨。

Qt 发给 Python 的请求快照

{
  "trace_id": "trace_qt_agent_xxx",
  "user_request": "帮我加工这个工件",
  "target_object_id": "feature_holes_001",
  "target_display_name": "孔组 A",
  "rejection_reason": ""
}

LLM 返回并经 Pydantic 校验的语义计划

{
  "schema_version": "1",
  "intents": [
    {
      "id": "intent_1",
      "intent": "machine_feature",
      "target": {
        "kind": "objects",
        "object_ids": ["feature_holes_001"]
      },
      "auto_generate_toolpath": true
    }
  ]
}

客户端执行拆解

  • SemanticIntentExecutor 读取 intent = machine_feature。
  • target 是 objects，直接得到 feature_holes_001。
  • 调用 BrowserConsole.machineFeature(["feature_holes_001"], true)。
  • FeatureRecognitionService 根据“holes / 孔”识别为 drilling。

最终状态变化

  • Repository 新增 op_drilling_feature_holes_001。
  • Repository 新增 toolpath_op_drilling_feature_holes_001。
  • BrowserTree 刷新出钻孔工序和钻孔刀轨。
  • CamViewport 显示钻孔刀轨预览。
案例 2：输入“帮我创建5个钻孔工序，3个型腔铣工序，4个平面铣工序” 目标：批量创建工序节点，但不自动生成刀轨。
1. Qt 收集 prompt

这个命令不一定依赖当前选择几何，target 可以为空。

2. LLM 拆数量

把一句自然语言拆成 create_operations 的 items 数组。

3. Pydantic 校验

operation_type 必须是 drilling / pocket / finishing，count 是整数。

4. Qt 解码

SemanticPlanDraftJsonCodec 得到 create_items。

5. 执行 intent

SemanticIntentExecutor 调用 BrowserConsole.createOperations。

6. 创建节点

OperationFactory 循环创建 12 个 operation，不调用 ToolPathService。

LLM 的任务拆解结果

{
  "schema_version": "1",
  "intents": [
    {
      "id": "intent_1",
      "intent": "create_operations",
      "items": [
        { "operation_type": "drilling", "count": 5 },
        { "operation_type": "pocket", "count": 3 },
        { "operation_type": "finishing", "count": 4 }
      ],
      "auto_generate_toolpath": false
    }
  ]
}

为什么不生成刀轨

  • 用户说的是“创建工序”，不是“创建工序并计算刀轨”。
  • 所以 LLM 设置 auto_generate_toolpath: false。
  • Qt 执行时只调用 OperationService.createOperation。
  • 后续用户可以手动右键计算，或再通过 Agent 要求计算。

客户端执行拆解

  • 遍历 items。
  • drilling 循环 5 次，pocket 循环 3 次，finishing 循环 4 次。
  • 每次通过 OperationCatalog 找默认参数和显示名。
  • 通过 OperationFactory 生成唯一 operation_id。

最终状态变化

  • BrowserTree 新增 12 个工序节点。
  • 每个 operation 都有自己的 object_id 和参数。
  • 不会新增 toolpath 节点。
  • CamViewport 不显示新刀轨，因为还没计算。
案例 3：输入“帮我打开树上的型腔铣工序，把刀具改大一些，再进行计算” 目标：复合任务拆成两个 intent：先编辑工序，再重新计算刀轨。
1. Qt 发送请求

prompt 里没有精确 ID，只有“树上的型腔铣工序”。

2. LLM 拆多意图

先 edit_operation，再 regenerate_toolpath，不生成组合 intent。

3. LLM 保留相对表达

刀具改大一些输出 larger_available_tool，不选最终 tool_id。

4. Qt 解析 target

query target 匹配 operation_type=pocket 的工序。

5. 必要时澄清

如果匹配多个型腔铣工序，HumanInLoopService 弹窗让用户选。

6. 执行并重算

ToolSelectionPolicy 选更大刀具；随后 ToolPathService 按 ref 重新计算。

LLM 拆出的 SemanticIntentPlan

{
  "schema_version": "1",
  "intents": [
    {
      "id": "intent_1",
      "intent": "edit_operation",
      "target": {
        "kind": "query",
        "object_type": "operation",
        "scope": "matching",
        "filters": { "operation_type": "pocket" }
      },
      "updates": [
        {
          "parameter": "tool_id",
          "value": null,
          "expression": "larger_available_tool"
        }
      ],
      "open_editor": true
    },
    {
      "id": "intent_2",
      "intent": "regenerate_toolpath",
      "target": {
        "kind": "ref",
        "ref": "previous.primary_object_ids"
      }
    }
  ]
}

LLM 做到哪一步

  • 识别“型腔铣”对应 operation_type = pocket。
  • 识别“刀具改大一些”对应 larger_available_tool。
  • 识别“再进行计算”需要第二个 regenerate_toolpath intent。
  • 不直接调用 C++，不直接决定 tool_016。

客户端执行拆解

  • resolveTarget(query) 查找 operation_type 匹配 pocket 的工序。
  • 如果多个匹配，进入 HumanInLoopService。
  • BrowserConsole.updateOperations 应用参数更新。
  • ParameterExpressionResolver 把 larger_available_tool 交给 ToolSelectionPolicy。
  • 上一 intent 的 primary_object_ids 被保存到执行上下文。
  • 第二个 intent 用 ref: previous.primary_object_ids 找到刚编辑的工序并重新计算。

最终状态变化

  • 例如当前刀具是 tool_010，更大刀具解析为 tool_016。
  • operation 的 tool_id 被更新。
  • 对应 operation_id 的 toolpath 被重新生成或更新。
  • BrowserTree 和 CamViewport 刷新，显示新的刀轨状态。

8. 当前边界规则

Console 规则

当前只有 BrowserConsole。它对应 BrowserTree / 工序树相关 UI 协调，不再引入 OperationConsole / ToolPathConsole / FeatureConsole。

Service 规则

OperationService、ToolPathService、FeatureRecognitionService、ToolSelectionPolicy 是业务服务，不是控制台。

LLM 规则

LLM 只输出 SemanticIntentPlan，不输出 Qt slots、C++ 方法名、Repository 操作或最终刀号。

刀具规则

Python 不读取 mock 刀具库；更大 / 更小刀具由 C++ ToolSelectionPolicy 解析为可用刀具。

目标规则

target 使用 objects / query / ref。current selection 是请求上下文，不是特殊目标类型。

多意图规则

复合需求拆成 intents[] 顺序执行，例如 edit_operation 后 regenerate_toolpath。

HumanInLoop

目标 query 不唯一时进入澄清弹窗，由用户选择具体对象，不和具体业务强绑定。

遗留规则

AgentWorkflowService 仍在代码里，但当前 Qt 主链路主要不走它，更像旧 roughing workflow 测试遗留。
